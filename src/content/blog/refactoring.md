리팩토링은 외부 동작을 바꾸지 않고 내부 구조를 개선하는 것이다. 핵심은 **기능 변경과 구조 개선을 동시에 하지 않는 것**이다.

## 언제 리팩토링하는가

타이밍이 전부다. 잘못된 시점의 리팩토링은 오히려 프로젝트를 망친다.

### 좋은 타이밍

- 새 기능을 추가하기 전, 추가할 영역이 이해하기 어려울 때
- 버그를 고친 직후, 왜 버그가 생겼는지 이해했을 때
- 코드 리뷰에서 구조적 문제를 발견했을 때

### 피해야 할 타이밍

- 데드라인 직전
- 테스트가 없는 코드 (테스트 먼저 작성할 것)
- 아직 요구사항이 안정되지 않은 기능

> 리팩토링은 기능이 안정된 뒤에 한다. 요구사항이 계속 바뀌는 코드를 리팩토링하면 두 번 일하는 꼴이다.

## 얼마나 리팩토링하는가

### 보이스카우트 규칙

코드를 건드릴 때마다 건드린 부분만 조금 더 깨끗하게 남긴다. 전체를 한 번에 고치려 하지 말고, 지나간 자리를 정돈하는 습관을 들인다.

### 범위 설정

범위를 명확히 하지 않으면 리팩토링이 끝없이 이어진다.

- PR 하나에 리팩토링 하나
- 기능 변경과 구조 변경을 같은 커밋에 섞지 않는다
- 변경 범위를 시작 전에 정해둔다

## 어떻게 리팩토링하는가

### 함수 분리

```python
# Before: 한 함수가 너무 많은 일을 함
def process_order(order):
    # 검증 20줄
    # 할인 계산 30줄
    # 저장 15줄

# After: 의도 단위로 분리
def process_order(order):
    validate_order(order)
    apply_discounts(order)
    save_order(order)
```

### 이름 개선

나쁜 이름은 코드를 읽을 때마다 뇌에 부하를 준다.

```python
# Before
def calc(x, y, z):
    return x * y * (1 - z)

# After
def calculate_discounted_price(price, quantity, discount_rate):
    return price * quantity * (1 - discount_rate)
```

### 중복 제거

같은 코드가 세 곳 이상 나타나면 추상화를 고려한다. 두 곳이라면 일단 지켜본다.

## 안전하게 리팩토링하기

#### 테스트 먼저

테스트 없이 리팩토링하면 동작이 바뀌었는지 알 수 없다. 테스트가 없다면 리팩토링 전에 작성한다.

#### 작은 단계

한 번에 큰 변경을 하면 어디서 깨졌는지 알기 어렵다. 작은 단계로 나눠서 각 단계마다 테스트가 통과하는지 확인한다.

#### 커밋 분리

리팩토링 커밋과 기능 변경 커밋은 반드시 분리한다. 나중에 `git blame`이나 `git bisect`로 추적할 때 혼란을 방지한다.

---

리팩토링의 목적은 코드를 예쁘게 만드는 게 아니라, **다음에 변경하기 쉽게** 만드는 것이다.
